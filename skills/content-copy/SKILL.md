# content-copy

A skill for writing SaaS sales and marketing copy that converts. It selects the right copywriting framework for the surface, gathers the context needed to write specifically (not generically), pulls language from real customers, leads with value before features, proves every claim, and ends with a clean editing pass.

The core idea: generic copy is a failure of intake, not a failure of writing. Spend the first move gathering enough context to write something only this product, for this audience, on this surface could say. Then write.

## When this skill fires

Trigger on any request to write or improve copy for:

- Landing page (hero, sections, or full page)
- Long-form sales page
- Cold outbound email or sequence
- Lifecycle / onboarding / activation email
- Paid ads (Google, Meta, LinkedIn, X)
- Pricing page
- In-product upsell, paywall, or expansion prompt
- Feature announcement (blog intro, in-app banner, changelog, launch post)
- Hero headlines, subheads, CTAs, microcopy

If the user provides existing copy and asks to "punch it up," "sharpen," "rewrite," or "make it convert," same skill applies — go straight to step 4 (the 4 C's pass) using the existing copy as the draft, but still confirm framework fit first.

## Step 1: Intake

Before writing a single line, gather the context that makes copy specific instead of generic. If the user already supplied any of these, don't re-ask — note what you have, then ask only for what's missing.

**Required context (ask if missing):**

1. **Surface** — Which of the surfaces above? Determines framework.
2. **Product** — One sentence: what the SaaS does.
3. **Audience** — Who specifically buys this? Role, company size, situation. "SMB owners" is too vague; "solo bookkeepers serving 5–20 restaurant clients" is usable.
4. **Primary outcome** — What does the audience get from the product? Frame as a real-world result, not a feature ("close books 4 days faster," not "automated reconciliation").
5. **Pain / status quo** — What's broken or annoying about how they do it today?
6. **Proof** — Anything credible to lean on: customer quotes, metrics, named logos, case studies, awards, integrations, security certs. If none exist, say so plainly — copy without proof has to work harder.
7. **CTA** — What action should the reader take? Sign up, book demo, start trial, upgrade, reply.

**Strongly preferred (ask, but proceed if user doesn't have them):**

8. **Voice of Customer (VoC)** — Review excerpts, sales call snippets, support tickets, G2 reviews, churn-survey quotes. The single biggest lever on whether copy resonates. Customer phrasing beats invented phrasing every time. Even 3–5 raw quotes change output quality dramatically.
9. **Brand voice** — Formal vs casual, witty vs earnest, technical vs plain. Default if unknown: clear, confident, plain — no jargon.
10. **Constraints** — Length limits, must-include phrases, banned words, SEO keywords, legal/compliance requirements.

If the user can't give you enough to write something specific, ask. A skipped intake is the fastest way to produce generic copy that nobody asked for.

## Step 2: Pick the framework

Match the surface to the framework(s) that work best for it. Mix when it helps — a long page often uses an AIDA-style hero, a PAS body section, JTBD-framed features, and PASTOR-style offer near the close.

| Surface | Primary framework | Notes |
|---|---|---|
| Landing page hero | **AIDA** or **StoryBrand opening** | Lead with the outcome, not the feature. |
| Long-form sales page | **PASTOR** or full **StoryBrand SB7** | Use the full arc; close with offer + risk reversal. |
| Cold email | **PAS** (short) | Empathy first, solution last. Under 90 words. |
| Lifecycle / onboarding email | **BAB** | One Before → After → Bridge per email. |
| Paid ads | **PAS** or **AIDA** + **FAB** headline | Match user's awareness stage. |
| Pricing page | **JTBD** + **FAB** + **StoryBrand stakes** | Frame tiers around jobs, not features. |
| In-product upsell / paywall | **BAB** + **JTBD** | Show the gap between current state and unlocked state. |
| Feature announcement | **BAB** + **FAB** | "Before this, you had to X. Now you can Y." |

### Framework definitions

**AIDA** — Attention (a hook that calls out the reader or names the outcome) → Interest (specifics that make them lean in) → Desire (proof + emotional payoff) → Action (clear, low-friction CTA).

**PAS** — Problem (name the pain in their language) → Agitate (make the cost of inaction real) → Solution (your product as the relief). Short, punchy, emotional.

**PASTOR** — Problem → Amplify (consequences, stakes) → Story / Solution → Transformation (the after state) → Offer (what they get, terms) → Response (specific CTA). Long-form, high-ticket.

**StoryBrand SB7** — Character (customer is the hero) → has a Problem → meets a Guide (your brand) → who gives them a Plan → and calls them to Action → that ends in Success or avoids Failure. Best when the product is complex and needs trust.

**BAB** — Before (current painful state) → After (the better state) → Bridge (your product is how they cross over). Punchy, great for emails and ads.

**FAB** — Feature (what it is) → Advantage (what it does better) → Benefit (what that means for the reader's life or job). Use to translate any feature into reader value.

**JTBD template** — "When I'm [situation], help me [job], so I can [desired outcome]." Use to frame every feature in outcome terms.

**4 C's** — Clear, Concise, Compelling, Credible. This is the editing pass at the end, not a writing framework.

## Step 3: Plan, then draft

Before writing the full piece, output a one-screen plan:

- Framework chosen (and why, in a line)
- The **spine**: the specific problem statement, the specific outcome claim, and the specific CTA
- The proof points you'll attach to each major claim
- Tone / voice notes

If you sense the intake left ambiguity (the audience is still fuzzy, the outcome is still abstract, there's no proof for a key claim), surface it here and get the user's input before drafting. Otherwise proceed to draft.

When drafting:

- **Lead with value, not functionality.** "Close your books in 4 days, not 14" beats "Automated reconciliation engine."
- **Use the customer's words.** If you have VoC, pull phrases verbatim. If not, write in plain language — no "best-in-class," no "robust solutions," no "leverage."
- **Attach proof to claims.** Every meaningful claim should have a number, a name, a quote, or a screenshot nearby. Vague claims without proof actively hurt credibility.
- **Specific beats clever.** Specific numbers, named outcomes, real timeframes. A clever headline that doesn't say anything loses to a specific headline that does.
- **One idea per section.** If a paragraph is doing two jobs, split it.
- **Match awareness stage.** Cold audiences need the problem named first. Warm audiences (already on your pricing page) need objection-handling and proof, not re-education.
- **Write CTAs as outcomes, not actions.** "See your team's first report" beats "Submit form." "Start your 14-day trial" beats "Sign up."

## Step 4: The 4 C's pass

Before delivering, run every line through this checklist. If a line fails any, rewrite it.

- **Clear** — Could a smart 14-year-old understand it on first read? No jargon they wouldn't already use. No clever phrasing that requires re-reading.
- **Concise** — Every word earns its place. Cut "in order to," "actually," "really," "very," "just," hedging clauses, and any sentence that could be a clause.
- **Compelling** — Does it make the reader feel the gap between now and after? If a section is neutral information without tension or payoff, it's filler.
- **Credible** — Every claim is either obvious, supported by proof, or removed. "10x faster" without a source is worse than "30% faster, measured across 2,400 customer migrations."

## Step 5: Deliver

Output in this order:

1. **The copy.** Cleanly formatted, ready to paste into a CMS or doc. Use markdown headers for sections (H1 for page title, H2 for major sections, etc.).
2. **Framework note.** One or two lines: which framework (or mix) you used and why.
3. **What's missing.** If the user couldn't give you VoC, proof, or specifics, list what would meaningfully improve the next pass. Don't apologize — just flag what would sharpen things.
4. **Variants (when useful).** For headlines, CTAs, and ad copy, offer 2–3 variants so the user can A/B or pick. For long-form pieces, skip this.

## Anti-patterns

Do not do these, regardless of what the user asks:

- **Generic openers.** "In today's fast-paced world..." Cut it.
- **Feature dumps as benefits.** "Our AI-powered platform..." is still a feature, not a benefit.
- **Unprovable superlatives.** "Best," "world-class," "industry-leading," "revolutionary." Replace with specifics or delete.
- **Unattached claims.** Any number, comparison, or outcome without a source.
- **Hedging.** "May help you potentially save up to..." commits to nothing. Pick a claim and stand behind it.
- **Cleverness over clarity.** A pun that obscures the value is worse than a plain sentence that delivers it.
- **Weak CTAs.** "Learn more" and "Submit" are the lowest-converting CTAs. Always name the outcome.
- **Buzzword salad.** "Synergize," "leverage," "ecosystem," "solution," "empower," "unlock." If a sentence reads like a LinkedIn post, rewrite it.

## On VoC

If the user has access to reviews, support tickets, sales call transcripts, or churn surveys and hasn't shared any, ask. Even 3–5 raw quotes change the quality of the output dramatically — they give you specific phrasing, the exact pains, and the exact words customers use for the outcome. If VoC isn't available, say so in the "What's missing" section of the delivery — it's the most useful thing the user could add to the next round.

## On iteration

Copy is rarely done in one pass. After the first draft, expect the user to react. Make one revision pass per round of feedback, and after each revision re-run the 4 C's checklist on whatever you changed — not just the new lines, since edits often introduce new clarity or credibility problems in the lines around them.
